# Technical Specification for resumer-scraper

## Scope and constraints

The repository shall be implemented in entity["software","Python","programming language"] and built around entity["software","Scrapy","Python web scraping framework"], with entity["software","PostgreSQL","relational database"] as the source of truth. That combination fits the requirement well: Scrapy’s spider model is built around source-specific crawl behaviour, its Item Loaders support per-field normalisation at extraction time, its Item Pipeline is intended for cleansing, validation, duplicate handling and database writes, and its JOBDIR mechanism can persist local crawl state for clean stop-and-resume runs. citeturn36view0turn36view2turn36view1turn36view4

The initial UK-only MVP should cover four upstreams: urlOutside IR35 Tech Jobsturn0search0, urlQuality Contractsturn13search0, the urlFantastic.jobs API docsturn0search6 exposed through urlRapidAPI pricing for Fantastic.jobsturn18search0, and the urlAdzuna developer docsturn4view0. A global uniqueness rule on canonical external URL is mandatory because source overlap is real, not hypothetical: Outside IR35 detail pages expose a source provider and an external Apply link, while Quality Contracts explicitly advertises multiple upstream providers, including urlLinkedInhttps://www.linkedin.com and urlFantastic.jobsturn0search6, so the same vacancy can surface through more than one origin. citeturn1view1turn14view0

One constraint materially changes the design. Adzuna’s public API terms allow publishing Adzuna listings and personal research, but state that other commercial, government or academic use is limited to a 14-day validation period unless licensed, and they explicitly say that API users must direct all queries via Adzuna and that contacting third-party content providers is a breach. That means the requirement to follow Adzuna redirect links to third-party job pages and scrape the downstream page text cannot be treated as a normal implementation detail; it is a legal and contractual blocker unless written permission or a separate commercial licence is obtained first. In the spec below, Adzuna support therefore has to be feature-flagged and disabled by default for any external-page fetch path. citeturn28view0

## Repository architecture

The repository should be organised as a small ingestion platform rather than a loose set of spiders. A clean shape is: `spiders/` for source adapters, `items/` for the common job item model, `loaders/` for field processors, `pipelines/` for dedupe and persistence, `services/` for URL canonicalisation, quota control and text cleaning, `storage/` for database access, `cli/` for local run commands, and `tests/` for fixtures, selector snapshots and integration tests. The key design rule is that every spider emits the same normalised item contract, and every source-specific quirk is handled either in the spider or in a clearly named adapter service rather than leaking into the rest of the codebase. That model matches how Scrapy expects spiders to define crawl behaviour and how Item Loaders reshape source data into a stable schema. citeturn36view0turn36view2

The runtime should have three distinct layers. The first layer is source acquisition, where spiders or API clients fetch list pages, detail pages and raw payloads. The second layer is synchronous ingest, where a shared Scrapy pipeline validates required fields, canonicalises URLs, resolves duplicates and writes raw and normalised records into Postgres. The third layer is post-ingest enrichment, which cleans descriptions, extracts structured fields for downstream matching, and enriches skills onto a controlled vocabulary. Scrapy’s own documentation is clear that item pipelines are sequential and intended for cleansing, validation, duplicate checks and persistence, which makes them a good fit for the ingest layer but a poor place for heavy enrichment jobs that would slow crawl throughput. citeturn36view1

For politeness and operational control, HTML sources and API sources should have different rate-control strategies. Website spiders should use Scrapy’s built-in pacing controls, with low per-domain concurrency and AutoThrottle enabled, because AutoThrottle adjusts request delay by latency and honours concurrency and delay limits. API sources should instead use explicit quota-aware middleware or a service-layer token bucket, because request-per-second and requests-per-month budgets are not problems AutoThrottle is designed to solve. citeturn36view3

Local pause-and-resume support should use JOBDIR only as a convenience layer for developer machines and ad hoc reruns. Scrapy’s own guidance says JOBDIR stores the state of a single spider run, can resume only when stopped cleanly, and must not be shared across spiders or across different jobs of the same spider. For that reason, persistent operational state such as stop markers, quota counters, crawl cursors and seen canonical URLs should live in Postgres, not in the filesystem. citeturn36view4

## Persistence and data model

Postgres should persist both the canonical job dataset and the operational state needed to keep the scraper safe and repeatable. The minimum schema should be centred on six tables.

- `jobs`: one canonical job row per normalised external URL. Core fields should include `external_url_normalized`, `external_url_raw`, `title`, `employer_name`, `recruiter_name`, `country_code`, `city`, `location_text`, `work_mode`, `ir35_status`, `rate_min`, `rate_max`, `rate_unit`, `currency`, `posted_at`, `description_raw`, `description_clean`, `source_quality`, `parser_confidence`, `first_seen_at`, `last_seen_at`, `status`, and `content_hash`.
- `job_occurrences`: one row per source listing URL. Fields should include `source_name`, `source_listing_url`, `source_provider`, `job_id`, `source_job_id`, `source_posted_at`, `first_seen_at`, `last_seen_at`, `crawl_run_id`, and a small JSON payload of source-specific metadata. This table is what fulfils the requirement to “add the source link to the job and to the job list” without creating duplicate jobs.
- `raw_payloads`: raw JSON, HTML, extracted text or response metadata for debugging, replay and later re-parsing. This can be keyed by `job_occurrence_id` plus payload type.
- `crawl_state`: per-source operational state such as last successful run time, stop markers, seen markers, next page cursor, hard-stop counters and any resume token needed by a source.
- `quota_state`: per-source counters for requests used, jobs used, billing-cycle start and end, accrued daily tokens, and last reset time.
- `job_enrichments`: structured extraction outputs such as normalised skills, category labels, contract length, tool stack, sectors and other fields intended for the downstream candidate-matching pipeline.

The canonical dedupe key shall be `external_url_normalized`, with a unique index on `jobs.external_url_normalized`. Inserts and updates should be implemented as upserts through entity["software","SQLAlchemy","Python SQL toolkit and ORM"] using PostgreSQL `ON CONFLICT`. That allows the scraper to insert a new job when the external URL is new, or to update freshness fields and attach a new occurrence when the external URL already exists. `job_occurrences` should also have a unique constraint on `(source_name, source_listing_url)` so that reruns are idempotent at the source-listing level as well. citeturn32view0

A listing that does not yet have a resolved external URL should not be promoted into `jobs`. Instead, it should remain in `job_occurrences` with `job_id = null` and `resolution_state = 'pending_external_url'`, with all raw artefacts stored in `raw_payloads`. That rule matters especially for sources such as Quality Contracts, where the static HTML clearly exposes rich job data but does not, from the static parse alone, expose the external destination in an obvious anchor tag, and for Adzuna, where redirect handling is legally constrained. This keeps the global uniqueness rule honest and avoids false merges based on weak heuristics.

Local development should use entity["software","Docker Compose","container orchestration tool"] to start Postgres automatically if it is not already running. The local entrypoint script should first check Docker availability via `docker version`, then check whether the named database container or service is already running via `docker ps --filter "name=..."`, and if not run `docker compose up -d db --wait`. The Compose file should define a Postgres healthcheck using `pg_isready`, and downstream services should depend on the database with `condition: service_healthy`. Data persistence should use a named Docker volume so that restart and rebuild cycles do not wipe the database unless the developer explicitly destroys volumes. PostgreSQL connection settings should be read from `DATABASE_URL` or from libpq-compatible environment variables such as `PGHOST`, `PGPORT`, `PGDATABASE` and `PGUSER`, so the script never hard-codes database coordinates. citeturn35view3turn35view4turn35view1turn35view0turn35view5turn35view6turn31view0

## Source ingestion behaviours

**Outside IR35.** The spider should start from the main jobs page on urlOutside IR35 Tech Jobsturn0search0, ignore sponsored cards, and paginate via the visible “Load more” link, which resolves to `?p=2` and can be followed repeatedly until there are no more pages. The crawl should stop when one of three conditions is met: the first previously seen non-sponsored listing URL is encountered, the feed ends, or the hard stop of 1,000 jobs is reached. Each detail page should extract the structured fields already present on-site: title, overview and requirement text, skills, location, work type, rate, recruiter, source provider and the external Apply URL. If `external_url_normalized` already exists in `jobs`, the pipeline should not create a second canonical job; it should only create or update the occurrence row linking the Outside IR35 listing to the existing job. citeturn9view3turn10view0turn1view1

**Quality Contracts.** The spider should use the today-only contract feed path described in the requirement and paginate by page number. The public jobs UI already exposes a date filter that includes “Today” and shows date sorting, and the listing view is interleaved with promotional and sign-up blocks, so selectors must only target real job cards. Quality Contracts also advertises multiple upstream providers, including urlLinkedInhttps://www.linkedin.com and urlFantastic.jobsturn0search6, which is strong evidence that duplicate vacancies across sources are expected. Its detail pages expose rate, IR35 status, working arrangement, city, country, a site-authored `Summary`, and a `Detailed Description From Employer`, so those should be preserved separately where possible instead of flattening them into one text blob. The stop strategy should persist a rolling set of the three newest previously seen listing URLs and stop once three stop markers have been hit, or when the feed ends, or when the hard limit is reached; that matches the requirement to tolerate sponsored placements and late arrivals. If the external destination can be resolved from the Apply action or embedded state, the listing may be promoted into `jobs`; if it cannot, it must stay staged in `job_occurrences` until an external URL resolver succeeds. Quality Contracts’ own terms also say job adverts and links depend on third-party information and linked websites are independent third-party sites, so the Quality Contracts URL should be treated as a source occurrence URL, not as the canonical job identity. citeturn14view0turn13search0turn17view1turn30view0

**Fantastic.jobs via RapidAPI.** The adapter should use the API rather than an HTML crawl. Fantastic.jobs states that its APIs refresh hourly, recommends regular polling and recommends loading jobs into a local database for latency and synchronisation reasons; it also advertises text descriptions, modified-job endpoints and expired-job endpoints. The RapidAPI listing says the LinkedIn Jobs API returns up to 100 jobs per request, supports filters including job title, description, location, company and remote mode, and allows location together with remote filtering. The repository config should therefore expose at least: `source_enabled`, `country=gb` or equivalent UK default, optional city, optional keyword, `remote_only`, `results_limit`, `offset`, `request_timeout`, and plan parameters for jobs-per-month, requests-per-month and burst rate. Rate limiting must be persisted in Postgres, not memory. The scheduler should compute daily budgets as `remaining_requests / remaining_days_until_cycle_end` and `remaining_jobs / remaining_days_until_cycle_end`, accrue fractional tokens daily, and only issue requests when both a request token and enough job tokens are available. This is essential because the Basic plan listed by RapidAPI is only 25 requests and 250 jobs per month, which works out to less than one request per day on a 30-day cycle; hourly polling is therefore impossible in development and should be disabled there even though the upstream data refreshes hourly. The production plan may use tighter second-level rate control, because RapidAPI lists 5 requests per second on Pro. citeturn1view5turn18search1turn21search0turn19search0turn19search2turn18search0

**Adzuna.** The adapter should use the official search endpoint in the UK market and request only recent jobs, sorted by date, with one-day age filtering where permitted by the endpoint parameters. Adzuna’s search response includes the listing `created` time and a `redirect_url`, but the documentation is explicit that the `description` field is only a snippet. That means the adapter can reliably stage metadata, but it cannot assume it has a complete job description from the API alone. However, the requirement to follow `redirect_url`, resolve the final external URL and scrape the downstream page text conflicts with Adzuna’s API terms, which say API users must direct all queries via Adzuna and that contacting third-party content providers is a breach. The only technically and contractually safe baseline spec is therefore: ingest Adzuna metadata into staging; persist the snippet, redirect URL and source metadata; disable external URL resolution and downstream page fetches by default; require an explicit legal/contract flag before enabling any redirect-following behaviour. If that approval exists later, the external fetch logic should still remain best-effort, low-rate and failure-tolerant, with raw text captured into Postgres for later reprocessing. citeturn4view0turn4view1turn28view0

## Cleaning and enrichment pipeline

The scraper should separate ingestion from enrichment. The synchronous ingest path should do only what is required to safely persist data: field validation, lightweight text trimming, URL canonicalisation, duplicate resolution and database writes. The heavier work—boilerplate removal, structured extraction, skill enrichment and quality scoring—should run after raw persistence in a worker or a second-stage job. That split is consistent with Scrapy’s pipeline model and keeps spiders fast, restartable and operationally simple. citeturn36view1

Cleaning rules should be source-aware. Outside IR35 pages contain sponsored cards on list pages; Quality Contracts interleaves sign-up prompts and also exposes both a generated summary and a longer employer description; Adzuna only gives a snippet in the search response. The cleaner therefore needs source-specific strip rules for navigation text, “Apply” boilerplate, sign-up prompts, duplicated paragraphs, recruiter disclaimers and other site chrome, while preserving the parts that actually matter to matching: responsibilities, requirements, skills, domain vocabulary, work arrangement, IR35 status, location and rate. For Quality Contracts in particular, the generated `Summary` and the `Detailed Description From Employer` should be stored separately if they can be parsed separately, because they are different provenance layers. citeturn9view3turn14view0turn17view1turn4view0

The structured output contract should include at least: canonical external URL, source listing URLs, title, employer, recruiter, posted date, country, city, free-text location, work mode enum, IR35 enum, contract type, contract duration where present, rate minimum, rate maximum, rate unit, currency, original skills text, normalised skills, description quality flag, and parser confidence. Skills enrichment should map synonyms and surface variants into a controlled vocabulary so that downstream candidate matching operates on stable skill IDs instead of a long tail of surface forms. Provider-enriched fields from Fantastic.jobs should be persisted in raw form, but the internal schema should remain repository-owned so that all four sources converge onto one comparable output shape. Fantastic.jobs’ own docs emphasise AI-enriched fields and modified or expired job endpoints, which is useful upstream capability but should not become the canonical internal model. citeturn1view5

The repository should also preserve change history. A practical design is either a `job_description_versions` table or a simple append-only snapshot table keyed by `job_id` and `content_hash`. That matters because jobs change, expire and reappear, and Fantastic.jobs explicitly advertises separate endpoints for modified and expired jobs. A version-aware model gives the downstream matcher a stable current view while preserving older payloads for debugging, model training and auditability. citeturn1view5

## Runtime, release, and acceptance criteria

Local developer flow should be one command: a CLI such as `resumer-scraper run <source>` should verify Docker availability, ensure the database service is running and healthy, apply migrations, and only then start the selected spider. The command should work whether the database is already running or not. The filesystem may hold logs and optional JOBDIR state for local reruns, but everything that matters to correctness—jobs, occurrences, raw payloads, stop markers and quotas—must survive in Postgres-backed persistence. Named volumes should preserve the database across ordinary Compose restarts, and deleting persisted state should require an explicit destructive action by the developer. citeturn35view3turn35view1turn35view0turn35view6

The Dockerfile should include the `org.opencontainers.image.source` label so the image is linked back to its repository. Publishing should be handled by urlGitHub Actionsturn33view1: a workflow triggered on `release.published` should build and push the image with `latest` and the release tag, while a separate `workflow_dispatch` workflow should build and push `dev` on demand. GitHub’s own documentation supports all of the required pieces here: pushing images to `ghcr.io`, authenticating with `GITHUB_TOKEN`, generating tags and labels with `docker/metadata-action`, and manually running workflows with inputs through `workflow_dispatch`. citeturn33view0turn33view1turn34view0turn33view2turn33view4
